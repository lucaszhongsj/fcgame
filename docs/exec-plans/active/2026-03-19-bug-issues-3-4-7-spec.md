# bug 标签问题修复规格（#3 / #4 / #7）

## 1. 背景与目标

当前仓库 `bug` 标签下有 3 个 open issue：

- #3 横屏适配问题（iPhone 横屏看不到游戏选择与控制按钮）
- #4 选择游戏后无法开始
- #7 无法操作第二位玩家

目标是在不引入构建系统的前提下，完成最小可验证修复，并补充可重复执行的自动化回归测试。

## 2. 范围

### 2.1 In Scope

- 修复 P2 键位映射导致的方向键失效问题（#7）
- 修复 ROM 加载流程在失败场景下误继续执行、提示不明确的问题（#4）
- 修复横屏小高度场景下 UI 被裁剪导致控制区不可见的问题（#3）
- 增加 Node 侧自动化回归测试（`node --test`）

### 2.2 Out of Scope

- 引入打包器、测试框架或后端服务
- 新增联机、存档、触屏双人等新功能
- 大规模重构 JSNES 内核

## 3. 问题定义与根因

### 3.1 Issue #7：第二位玩家无法操作

已确认 `source/keyboard.js` 的 P2 方向键分支误用了 `87/83/65/68`（W/S/A/D）并写入 `state1`，导致：

- P2 方向键（↑↓←→）不会生效
- P1/P2 输入逻辑存在冲突风险

### 3.2 Issue #4：选择游戏后无法开始

`source/ui.js` 的 `loadROM` 当前实现存在两个鲁棒性缺陷：

- 使用 `complete` 回调，无论请求成功或失败都会继续 `loadRom` + `start`
- URL 使用 `escape(...)`，对路径字符过度编码，兼容性不稳定

在 ROM 请求失败（404、file 协议限制、服务器路径不匹配）时，用户会看到“无法开始”但缺少明确原因与指引。

### 3.3 Issue #3：横屏看不到选择与功能按钮

`css/jsnes.css` 仅有空的 landscape 媒体查询，且 `.screen` 使用 `overflow: hidden`。在 iPhone 横屏低高度场景下，`nes-roms` 和 `nes-controls` 容易被裁剪不可见。

## 4. 方案设计

### 4.1 #7 修复方案

- 将 P2 方向键映射改为标准方向键 keyCode（`38/40/37/39`）
- 保持 P2 A/B/Select/Start 现有键位（Num1/Num2/Num/Num*）
- 增加单元测试覆盖 P2 方向键按下/抬起状态变化

### 4.2 #4 修复方案

- `loadROM` 改为显式 `success/error` 分支：
  - `success`：解析并加载 ROM，成功后启动并 enable 按钮
  - `error`：更新状态文本，给出可操作提示，不进入启动流程
- 请求 URL 直接使用 ROM 原始相对路径（不再使用 `escape`）
- 增加 `file://` 场景提示（建议通过本地 HTTP 服务启动）
- 增加回归测试：
  - 成功与失败状态文案生成逻辑
  - 协议判断逻辑（`file:`）

### 4.3 #3 修复方案

- 补充横屏低高度媒体查询（`orientation: landscape` + `max-height`）
- 保证 `nes-roms`、`nes-controls` 可见：
  - 允许竖向滚动或减小视觉占用
  - 缩小横屏下控件尺寸并优化间距
- 增加 CSS 回归测试，锁定关键规则存在

## 5. 验收标准

- #7：桌面端按 `↑↓←→` 可控制 P2 方向，按键释放后状态恢复
- #4：
  - ROM 请求失败时不再误启动模拟器
  - 状态区出现明确失败提示
  - `file://` 打开时给出启动方式指引
- #3：在横屏低高度视口下，游戏选择框与暂停/重启/音效/缩放按钮可见且可操作
- `node --test` 全部通过

## 6. 风险与回滚

- 风险：不同浏览器对本地协议与键盘码行为细节存在差异
- 回滚策略：按 issue 维度回滚对应文件改动（互不耦合）
